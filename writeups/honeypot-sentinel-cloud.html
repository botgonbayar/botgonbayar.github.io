Building a Cloud SIEM Honeypot: Azure Sentinel, Live Attack Telemetry, and an AI Triage Layer

I put a deliberately exposed Windows box on the public internet, wired it into Microsoft Sentinel, and watched real attackers show up within minutes. Then I went past the tutorial: sharper detections, MITRE mapping, and an AI layer that turns raw alerts into analyst-ready summaries.

A coworker once asked me whether a brute-force alert on a server was something to worry about, and I realized I had never actually watched the other side of that interaction happen in real time. I knew the theory: failed logon events, source IPs, the whole MITRE vocabulary. But I had never stood up a target, opened the front door, and let the internet do what it does. So I did exactly that, and turned it into a small SOC pipeline.

This guide walks through the entire build step by step: a deliberately exposed Windows VM in Azure, Microsoft Sentinel ingesting its security events, custom KQL detections mapped to MITRE ATT&CK, and an AI triage layer that summarizes every incident before an analyst opens it. The foundation is the Logos Red Azure Sentinel lab. I followed it end to end, then kept going, because the tutorial stops at a single string-match query and a real detection environment is a different animal. Steps 1 through 5 are the base build; Steps 6 through 8 are where it becomes something closer to a working SOC workflow.

Lab only. We are about to put a wide-open Windows box on the public internet. That is the entire experiment, and it is only safe because the VM holds nothing, lives in its own resource group, and gets deleted when we're done. Expose the throwaway, never the real thing.

What You'll Need#

Accounts & Software

  • An Azure account - which means a credit card is needed as well (the free tier with $200 in credits is plenty)
  • A web browser. Everything in this lab happens in the Azure portal
  • Remote Desktop client (built into Windows, free on macOS) to test logons
  • Optional, for Step 8: access to Azure OpenAI or an OpenAI API key for the AI triage layer

Knowledge

  • Basic familiarity with the Azure portal helps but isn't required; every click is shown
  • No prior KQL experience needed. Every query is provided and explained

Cost heads-up before we start. Sentinel bills on data ingestion, and a wide-open VM generating thousands of events an hour adds up faster than you'd guess. Deallocate the VM whenever you're not actively collecting, and delete the whole resource group when you're done. My total across about three days of collection was a few dollars. Leave it running and forget it, and that number stops being funny.

Step 1: Create the Honeypot VM#

First we build the bait: a plain Windows VM in its own dedicated resource group.

  1. Sign in to the Azure portal and go to Virtual machines → Create → Azure virtual machine
  2. Under Resource group, click Create new and name it something obvious, e.g. rg-honeypot-lab

    3. Why a dedicated resource group? When the lab is over, you delete the group and everything inside it goes with it: VM, disk, NIC, public IP. No orphaned resources quietly billing you.

  3. Name the VM (I used Secora-CORP-0241) because if you know me from my Active Directory lab writeup, Secora is the fictional company I based the lab off of, so to make the VM sound more enticing to the public, I formatted the name to resemble a legitimate company asset, and then pick a region close to you. Remember which one; the workspace must match it later
  1. For Image, make sure you choose Windows 10 Enterprise, version 22H2, since you will get this error if you try to proceed with for example, Windows 11 images: "This virtual machine requires a subscription without any spending limit or temporary payment method set. Use a different subscription or update your subscription (requires account administrator privileges) at [Account Center](https://account.windowsazure.com) to continue."
  2. For Availibility options, make sure you pick No infrastructure redudancy required because choosing avalibility zones for our VMs will restrict certain sizes
  3. For Size, pick something small but not tiny, around 2 vCPUs and 8 GB RAM is perfect in my opinion (e.g. Standard_D2s_v3)

    4. Why not the smallest size? The Azure Monitor Agent that ships events to Sentinel has real overhead. Starve the VM and it drops events during I/O spikes, which means you miss the exact bursts of attack activity you built this lab to see.

  4. Set a username and password. Write them down; you'll RDP in later to generate test events
  5. Under Inbound port rules, leave RDP (3389) selected
  6. Checkmark "I confirm I have an eligible Windows 10/11 license with multi-tenant hosting rights.", then Review + Create, then Create and let it deploy
  7. Deployment might take a couple minutes to finish but eventually you'll end up on a screen that references the screenshot below

Step 2: Open It to the Internet#

This is the step that makes it a honeypot. By default Azure gives you a sensible Network Security Group that only allows RDP. We're going to replace it with a rule that allows everything.

  1. Open your VM and go to Networking → Network settings
  2. Delete the default RDP inbound rule
  3. Click Create port rule → Inbound port rule and set:
    • Source: Any
    • Source port ranges: *
    • Destination: Any
    • Destination port ranges: *
    • Protocol: Any · Action: Allow
    • Priority: 100
    • Name: ALLOW-ALL-INBOUND
  4. Click Add

Why priority 100? Lower priority numbers win in Azure, so 100 guarantees this allow-all rule is evaluated before any default deny. An allow-all inbound rule on a public IP is exactly what you spend your career preventing; it belongs on this disposable VM and nowhere near anything real.

Step 3: Deploy Microsoft Sentinel#

The thing worth internalizing here is that Sentinel doesn't actually store your logs; the Log Analytics Workspace does. Sentinel is the analytics and alerting brain that sits on top of it. So we create the workspace first, then enable Sentinel on it.

  1. In the portal search bar, search for Log Analytics workspaces → Create
  2. Put it in the same resource group as the VM, name it (e.g. law-honeypot), and pick the same region as your VM
  3. Click Review + create → Create

Why the same region? It keeps you out of cross-region data-transfer charges, which is a small thing until your honeypot is ingesting thousands of events an hour.

  1. Now search for Microsoft Sentinel → Create
  2. Select the workspace you just made and click Add

Step 4: Connect the VM to Sentinel#

Sentinel can't analyze events it never receives. The bridge is a data connector plus a Data Collection Rule.

  1. In Sentinel, go to Content hub, search for Windows Security Events, and click Install
  2. Refresh the Sentinel page after the install finishes

The refresh matters. The connector won't appear in the list until you reload the page, and that missing refresh is the single most common reason people think their setup is broken when it isn't.

  1. Go to Configuration → Data connectors, open Windows Security Events via AMA, and click Open connector page
  2. Click Create data collection rule
  3. Name it, then on the Resources tab check your honeypot VM
  4. On the Collect tab, choose All Security Events
  5. Click Review + create → Create

Why All Security Events instead of Common? "Common" drops some of the event IDs that matter for lateral-movement and persistence detection, and for a lab you want the full firehose. In production you'd filter aggressively here to control ingestion cost, but that's a production problem, not a lab one.

Give it five to ten minutes. The agent installs as a VM extension after the rule is created, and there's an initialization window where queries return nothing. If you query immediately and panic, you'll waste an hour troubleshooting a thing that just needed to finish booting.

Step 5: Run Your First KQL Query#

Sentinel queries are written in KQL, Kusto Query Language. It's Microsoft's own thing, close enough to SQL that you can read it on sight. Let's prove data is flowing.

  1. RDP into the VM once using its public IP, so there's at least one logon event to find
  2. In Sentinel, go to Logs, close the query gallery, and run:
SecurityEvent
| where Activity contains "success"

The base query: fine for a first look, far too broad for an alert.

You should see your own logon in the results. The problem is the breadth: matching the string success in the Activity field catches logoffs and a pile of benign events alongside the real logons. It tells you data exists; it does not tell you anything an analyst would act on. This is where the original tutorial stops, and where we keep going.

Step 6: Write Sharper Detections#

Real detections key on event IDs, which are specific, stable, and fast. Event ID 4625 is a failed logon. A single one is meaningless; ten from the same IP in ten minutes is a campaign. Run each query below in Logs and you'll watch the noise resolve into signal.

6.1: Brute-force attempts (failed logons over a threshold)

// Brute-force signature: rapid failed logons (Event ID 4625) per source IP
let threshold = 10;
SecurityEvent
| where EventID == 4625
| where TimeGenerated > ago(1h)
| summarize
    FailedAttempts   = count(),
    FirstAttempt     = min(TimeGenerated),
    LastAttempt      = max(TimeGenerated),
    AccountsTargeted = dcount(TargetAccount)
    by IpAddress, Computer
| where FailedAttempts >= threshold
| extend DurationMin = datetime_diff('minute', LastAttempt, FirstAttempt)
| project IpAddress, Computer, FailedAttempts, AccountsTargeted,
          FirstAttempt, LastAttempt, DurationMin
| sort by FailedAttempts desc

Counting failures per IP and only surfacing the ones past a threshold.

6.2: Brute-force success (the one that actually means trouble)

The higher-fidelity detection is a successful logon (4624) from an IP that was just throwing failures at you. That's a brute-force that worked. Joining the two event streams turns a noisy signal into a precise one, and the MITRE technique is annotated right in the query so the mapping travels with the alert. 

// Brute-force SUCCESS: a 4624 logon from an IP that was just failing (4625)
let lookback = 1h;
let failThreshold = 5;
let bruteForceIPs =
    SecurityEvent
    | where TimeGenerated > ago(lookback)
    | where EventID == 4625
    | summarize FailCount = count() by IpAddress
    | where FailCount >= failThreshold
    | project IpAddress;
SecurityEvent
| where TimeGenerated > ago(lookback)
| where EventID == 4624
| where LogonType == 10          // 10 = RemoteInteractive (RDP)
| join kind=inner bruteForceIPs on IpAddress
| project TimeGenerated, Computer, IpAddress, TargetAccount, AuthenticationPackageName
| extend MitreTactic    = "Initial Access",
         MitreTechnique = "T1110.001 - Brute Force: Password Guessing"
| sort by TimeGenerated desc

The success-after-failure pattern: low volume, high meaning.

Why filter on LogonType 10? It scopes the match to RDP sessions specifically, so you're not alerting on a local console logon that happens to follow some unrelated failures. Small filter, large difference in false-positive rate.

6.3: Post-compromise activity

Three more event IDs describe an attacker who is already in and trying to stay there or cover their tracks. In a honeypot those are gold, because reaching them means the brute-force succeeded and someone is now operating on the box. 

// 4732: account added to a privileged local group (privilege escalation)
SecurityEvent
| where EventID == 4732
| where TargetAccount contains "Administrators"
| project TimeGenerated, Computer, SubjectAccount, MemberName, TargetAccount
| extend MitreTactic = "Privilege Escalation", MitreTechnique = "T1078 - Valid Accounts"

// 4719: audit policy changed (defense evasion / log tampering)
SecurityEvent
| where EventID == 4719
| project TimeGenerated, Computer, SubjectAccount, AuditPolicyChanges
| extend MitreTactic = "Defense Evasion", MitreTechnique = "T1562.002 - Disable Windows Event Logging"

// 4698: scheduled task created (persistence)
SecurityEvent
| where EventID == 4698
| project TimeGenerated, Computer, SubjectAccount, TaskName
| extend MitreTactic = "Persistence", MitreTechnique = "T1053.005 - Scheduled Task"

Three post-compromise detections the base tutorial never reaches for.

Step 7: Turn Queries into Alert Rules#

A query you run by hand is a hunt. A query Sentinel runs on a schedule is a detection. Let's promote them.

  1. In Logs, with your query open, click New alert rule → Create Microsoft Sentinel alert
  2. Name the rule and set a severity (see the tiering below)
  3. Under MITRE ATT&CK, tag the matching tactic: Initial Access for the brute-force success rule, Persistence for the scheduled-task rule, and so on
  4. On the rule logic tab, set Run query every 5 minutes with Lookup data from the last 1 hour
  5. Under Entity mapping, map IpAddress to an IP entity and TargetAccount to an Account entity
  6. Under Incident settings, enable grouping: group alerts by IP entity within a 1-hour window
  7. Save, then repeat for each detection from Step 6

The grouping setting is the sanity-saver. A single determined IP can trip a brute-force rule hundreds of times in an hour. Grouped by source IP and a time window, that becomes one incident with a count, instead of a screen full of identical noise that buries anything real.

Here's everything that changed between the base tutorial and what we just built:

AreaBase tutorialWhat we ran instead
Detection logicString match on "success"Event-ID based (4624 / 4625 / 4719 / 4732 / 4698) with thresholds
SeverityOne level for everythingTiered: brute-force Medium, post-brute success High, policy change Critical
MITRE mappingSingle tactic in the UITactic + technique embedded in the KQL so it rides into the incident
Lookback window5-min run, 5-min lookback5-min run, 1-hour lookback, which catches slow campaigns under the per-window bar
Entity mappingNot configuredIP and Account mapped as entities, enabling the investigation graph and UEBA
Incident groupingDefault (one per alert)Grouped by IP + 1-hour window, so one attacker can't cause an alert storm
AutomationNoneLogic App playbook: geo + AI enrichment on incident creation (Step 8)

Step 8: Add the AI Triage Layer#

This is where the project stops being a lab build and starts being something I'd actually want at a SOC desk. The most tedious part of Tier-1 work is reading raw event data and reconstructing, in your head, the story of what happened. That reconstruction is exactly the kind of thing a language model is good at, so we put one in the pipeline.

8.1: Create the playbook

  1. In Sentinel, go to Automation → Create → Playbook with incident trigger
  2. Name it (e.g. pb-ai-triage) and open it in the Logic Apps designer
  3. Add steps in this order:
    • Entities - Get IPs and Entities - Get Accounts (Sentinel connector)
    • An HTTP action calling a geo/ASN lookup for the attacker IP
    • An HTTP action calling the AI model with the payload below
    • Add comment to incident, posting the model's response

The body of the AI call looks like this:

{
  "model": "gpt-4o",
  "messages": [
    {
      "role": "system",
      "content": "You are a SOC analyst assistant. Given a Microsoft Sentinel incident, produce a triage summary in three sections: 1) What happened (2-3 sentences), 2) MITRE ATT&CK mapping with technique IDs, 3) Recommended immediate actions. Write for a Tier 2 analyst."
    },
    {
      "role": "user",
      "content": "Incident: @{triggerBody()?['object']?['properties']?['title']}\n\nEntities:\n@{variables('EntitySummary')}\n\nGeo:\n@{variables('GeoEnrichment')}"
    }
  ],
  "max_tokens": 500,
  "temperature": 0.1
}

The Logic App's HTTP action body. Low temperature keeps the output factual and consistent.

  1. Back in Automation, create an Automation rule: when an incident is created, run pb-ai-triage

8.2: Test the prompt before wiring it in

I'd recommend iterating on the prompt outside Logic Apps first. The same call as a standalone Python script against Azure OpenAI: 

import os
from openai import AzureOpenAI

client = AzureOpenAI(
    api_key=os.environ["AZURE_OPENAI_KEY"],
    api_version="2024-02-01",
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
)

def triage(incident_title: str, entities: dict) -> str:
    """Generate an analyst-ready triage summary for a Sentinel incident."""
    summary = "\n".join([
        f"- IP: {entities.get('ip', 'N/A')}",
        f"- Account: {entities.get('account', 'N/A')}",
        f"- Host: {entities.get('host', 'N/A')}",
        f"- Country: {entities.get('country', 'N/A')} ({entities.get('asn', 'N/A')})",
        f"- Failed attempts: {entities.get('failed', 0)}",
    ])

    resp = client.chat.completions.create(
        model="gpt-4o",                       # your Azure deployment name
        messages=[
            {"role": "system", "content": (
                "You are a SOC analyst assistant. Produce a triage summary with "
                "three sections: What happened (2-3 sentences), MITRE ATT&CK mapping "
                "(tactic + technique ID), and three recommended actions. Be technical "
                "and precise; assume a Tier 2 audience."
            )},
            {"role": "user", "content": f"Incident: {incident_title}\n\nEntities:\n{summary}"},
        ],
        max_tokens=500,
        temperature=0.1,
    )
    return resp.choices[0].message.content


if __name__ == "__main__":
    print(triage(
        "Brute Force Success: RDP (T1110.001)",
        {"ip": "185.220.101.47", "account": "Administrator", "host": "honeypot-vm-01",
         "country": "Netherlands", "asn": "Tor exit node", "failed": 347},
    ))

The same triage call as a standalone script, useful for iterating on the prompt.

8.3: Enrich the IP before the model sees it

A raw IP is almost useless to an analyst, so before the triage call I run the IP through a geo/ASN lookup and a Tor/VPN/datacenter classification, then let the model interpret what that combination implies about the actor. A residential IP in one country reads very differently from a datacenter range or a known Tor exit. 

import requests

def enrich_ip(ip: str, client) -> dict:
    """Geo + ASN lookup, then an AI read on what the combination suggests."""
    geo = requests.get(
        f"http://ip-api.com/json/{ip}"
        "?fields=country,regionName,isp,org,as,proxy,hosting",
        timeout=5,
    ).json()

    context = (
        f"IP: {ip}\nCountry: {geo.get('country')}\n"
        f"Org: {geo.get('org', geo.get('isp'))}\nASN: {geo.get('as')}\n"
        f"Proxy/VPN: {geo.get('proxy')}\nHosting/DC: {geo.get('hosting')}"
    )

    read = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": (
                "You are a threat-intel analyst. In 2-3 sentences, assess likely "
                "actor type (opportunistic scanner, targeted, botnet node), "
                "infrastructure type, and OPSEC posture. Be concise."
            )},
            {"role": "user", "content": context},
        ],
        max_tokens=200, temperature=0.2,
    )
    return {"geo": geo, "assessment": read.choices[0].message.content}

IP enrichment feeding an AI actor-context read, which then feeds the triage prompt.

The model advises; a human pulls the trigger. The playbook can also ask the model which containment action fits (isolate the host, block the IP at the NSG, watchlist it, or do nothing) and send that recommendation for approval before anything executes. For a lab that's slightly theatrical, but it's the right shape for the real thing: the automation never takes an irreversible action on its own.

What Actually Showed Up#

Then I turned it on and waited, which did not take long. The first brute-force attempt against the box landed inside fifteen minutes of it going public. Internet-wide scanners find a fresh public IP almost immediately, and there is something genuinely sobering about watching the first failed-logon events roll in before you've finished your coffee.

Over the next day the failed-logon traffic came from dozens of countries, with the usual suspects dominating the volume: hosting ranges, Tor exits, and a long tail of clearly compromised residential hosts being used as relays. The targeted usernames were exactly the boring, predictable set of Administrator, admin, user, test, and guest, because automated tools cycle through the obvious ones first, every time. Volume peaked overnight in my timezone, which lines up with automated campaigns running during business hours elsewhere in the world.

The two things I'll keep from running this: event-ID detections produced zero false positives from the benign logoff events that the tutorial's string-match would have flagged, and the AI triage comment turned each incident from a four-to-six-minute manual read into something I could grasp in well under a minute, because the narrative was already sitting there when I opened it. Neither of those is a dramatic result on its own. Together they're the difference between a lab and a workflow.

What We've Accomplished#

  • Deployed an internet-exposed Windows honeypot VM in an isolated Azure resource group
  • Built the full telemetry pipeline: AMA agent → Data Collection Rule → Log Analytics → Sentinel
  • Wrote event-ID-based KQL detections for brute force, brute-force success, privilege escalation, log tampering, and persistence
  • Promoted them to analytics rules with MITRE ATT&CK mapping, entity mapping, tiered severity, and incident grouping
  • Automated incident enrichment with a Logic App playbook: geo/ASN context plus an AI-written triage summary on every new incident
  • Collected and analyzed real attack traffic from the open internet

Tear it down. When you're done collecting, go to Resource groups, open rg-honeypot-lab, and click Delete resource group. The honeypot's job is to exist briefly, teach you something, and disappear.

What's Next#

This is the first piece of a longer cloud-security thread I'm working through. The geo data this lab already collects wants to become a live attack map in a Sentinel Workbook. Watching the failed logons light up a world map in real time is the obvious next move, and the enrichment is half-built for it already. After that I want to wire MISP into Sentinel so attacker IPs get cross-referenced against known threat-actor infrastructure automatically, and layer Defender for Cloud on top to capture what actually happens on the box after a brute-force succeeds, rather than inferring it from event IDs.

The thing this lab drove home, though, is smaller and more durable than any of that. You can read about RDP brute-force for years and it stays abstract. Open one real port to the internet, watch the attempts arrive inside fifteen minutes, and it stops being abstract permanently. Build the detection around what you watched, not around what the tutorial told you to match, and the queries get sharper because you've seen the thing they're meant to catch.
