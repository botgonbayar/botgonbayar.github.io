Building a Cloud SIEM Honeypot: Azure Sentinel, Live Attack Telemetry, and an AI Triage Layer

I put a deliberately exposed Windows box on the public internet, wired it into Microsoft Sentinel, and watched real attackers show up within minutes. Then I went past the tutorial — sharper detections, MITRE mapping, and an AI layer that turns raw alerts into analyst-ready summaries.

BO
Bat Otgonbayar
Systems & Security · LinkedIn

A coworker once asked me whether a brute-force alert on a server was something to worry about, and I realized I had never actually watched the other side of that interaction happen in real time. I knew the theory — failed logon events, source IPs, the whole MITRE vocabulary — but I had never stood up a target, opened the front door, and let the internet do what it does. So I did exactly that, and turned it into a small SOC pipeline.

The foundation here is the Logos Red Azure Sentinel guide: spin up a Windows VM, expose it, and feed its security events into Microsoft Sentinel. I followed it end to end, got it working, and then kept going — because the tutorial stops at a single string-match query, and a real detection environment is a different animal. This writeup is the version I wish I'd had: the base build, the places I tightened it, and the AI layer I bolted onto the incident pipeline afterward.

One principle up front, the same one that keeps a honeypot a lab and not a liability. Expose the throwaway, never the real thing. The whole point is to attract noise onto a machine that holds nothing, lives in its own resource group, and gets deleted when I'm done. I'll flag the spot where that distinction actually costs money if you ignore it.

The build at a glance#

Five moving parts, and each one only matters because of the one after it.

  1. A Windows VM in Azure with a Network Security Group deliberately set to allow everything inbound. This is the honeypot — the bait.
  2. The Azure Monitor Agent on that VM, shipping Windows Security Events into a Log Analytics Workspace through a Data Collection Rule.
  3. Microsoft Sentinel sitting on top of the workspace, running KQL analytics rules that turn raw events into alerts and incidents.
  4. A set of detections that go past "someone logged in" — brute-force thresholds, post-brute-force success, privilege escalation, and persistence, each mapped to MITRE ATT&CK.
  5. A Logic App playbook that enriches every new incident with geo data and an AI-written triage summary before an analyst ever opens it.

Now the detail, in the order I built it.

Standing up the victim VM#

Once you're signed into the Azure portal, the VM itself is the easy part — the interesting decisions are in the networking. Start from Virtual machines → Create → Azure virtual machine, and give it a fresh resource group of its own. Keeping the whole lab in a dedicated resource group is not cosmetic: when you're done, you delete the group and everything goes with it, no orphaned disks quietly billing you.

For the image, pick Windows 10 or 11. For size, anything cheap works, but I'd aim for roughly two vCPUs and a few gigs of RAM — the Azure Monitor Agent that ships events to Sentinel has real overhead, and if you starve the VM it drops events during I/O spikes, which means you miss the exact bursts of activity you built this to see. Set a username and password, keep RDP enabled, and leave the disk settings at their defaults.

The networking step is the one that makes this a honeypot. By default Azure gives you a sensible Network Security Group that only allows RDP. Delete that rule and replace it with one that allows everything: source *, destination port range *, priority 100. Lower priority numbers win in Azure, so 100 guarantees this "allow all" rule is evaluated before any default deny.

Lab only. An allow-all inbound rule on a public IP is exactly what you spend your career preventing. It belongs on a disposable VM in an isolated resource group and nowhere near anything real. The reason we open it this wide is so the box lights up with attack traffic we can study — that's the entire experiment.

Review, create, and let it deploy. While the VM spins up, we move to Sentinel.

Wiring up Sentinel#

The thing worth internalizing here is that Sentinel doesn't actually store your logs — the Log Analytics Workspace does. Sentinel is the analytics and alerting brain that sits on top of it. So the order matters: search for Microsoft Sentinel, create it, and point it at a workspace in the same region as your VM. Same region keeps you out of cross-region data-transfer charges, which is a small thing until your honeypot is ingesting thousands of events an hour.

With Sentinel created, go to Content hub and install the Windows Security Events solution. That pack brings in the AMA-based data connector, plus pre-built workbooks and analytics rules you'll use later. Refresh the Sentinel page afterward — the connector won't appear in the list until you do, and that missing refresh is the single most common reason people think their setup is broken when it isn't.

Open the Windows Security Events via AMA connector, create a new Data Collection Rule, and attach your VM to it. When it asks which events to collect, choose All Security Events rather than Common. "Common" drops some of the event IDs that matter for lateral-movement and persistence detection, and for a lab you want the full firehose. In production you'd filter aggressively here to control ingestion cost — but that's a production problem, not a lab one.

Give it five to ten minutes. The agent installs as a VM extension after the DCR is created, and there's an initialization window where queries return nothing. If you query immediately and panic, you'll waste an hour troubleshooting a thing that just needed to finish booting.

The first query, and why it's not enough#

Sentinel queries are written in KQL, Kusto Query Language — Microsoft's own thing, close enough to SQL that you can read it on sight. The tutorial's starting query is genuinely the right first move, because it proves data is flowing: 

SecurityEvent
| where Activity contains "success"

The base query — fine for a first look, far too broad for an alert.

Run that after you've RDP'd in once, and you'll see your own logon. The problem is the breadth: matching the string success in the Activity field catches logoffs and a pile of benign events alongside the real logons. It tells you data exists; it does not tell you anything an analyst would act on. So the first thing I did was throw it away and write detections around event IDs, which are specific, stable, and fast.

Sharper detections — the part the tutorial skips#

Here's the first real detection: rapid failed logons from a single source, which is the signature of an RDP brute-force. Event ID 4625 is a failed logon. A single one is meaningless; ten from the same IP in ten minutes is a campaign. Thresholds are what separate an alert from a notification nobody reads. 

// Brute-force signature: rapid failed logons (Event ID 4625) per source IP
let threshold = 10;
SecurityEvent
| where EventID == 4625
| where TimeGenerated > ago(1h)
| summarize
    FailedAttempts   = count(),
    FirstAttempt     = min(TimeGenerated),
    LastAttempt      = max(TimeGenerated),
    AccountsTargeted = dcount(TargetAccount)
    by IpAddress, Computer
| where FailedAttempts >= threshold
| extend DurationMin = datetime_diff('minute', LastAttempt, FirstAttempt)
| project IpAddress, Computer, FailedAttempts, AccountsTargeted,
          FirstAttempt, LastAttempt, DurationMin
| sort by FailedAttempts desc

Counting failures per IP and only surfacing the ones past a threshold.

The higher-fidelity detection is the one that actually means trouble: a successful logon (4624) from an IP that was just throwing failures at you. That's a brute-force that worked. Joining the two event streams turns a noisy signal into a precise one, and I annotate the MITRE technique right in the query so the mapping travels with the alert into the incident. 

// Brute-force SUCCESS: a 4624 logon from an IP that was just failing (4625)
let lookback = 1h;
let failThreshold = 5;
let bruteForceIPs =
    SecurityEvent
    | where TimeGenerated > ago(lookback)
    | where EventID == 4625
    | summarize FailCount = count() by IpAddress
    | where FailCount >= failThreshold
    | project IpAddress;
SecurityEvent
| where TimeGenerated > ago(lookback)
| where EventID == 4624
| where LogonType == 10          // 10 = RemoteInteractive (RDP)
| join kind=inner bruteForceIPs on IpAddress
| project TimeGenerated, Computer, IpAddress, TargetAccount, AuthenticationPackageName
| extend MitreTactic    = "Initial Access",
         MitreTechnique = "T1110.001 - Brute Force: Password Guessing"
| sort by TimeGenerated desc

The success-after-failure pattern — low volume, high meaning.

Filtering on LogonType == 10 is the detail that keeps this honest: it scopes the match to RDP sessions specifically, so you're not alerting on a local console logon that happens to follow some unrelated failures. Small filter, large difference in false-positive rate.

Beyond the login story, three more event IDs are worth watching because they describe an attacker who is already in and trying to stay there or cover their tracks. In a honeypot those are gold, because reaching them means the brute-force succeeded and someone is now operating on the box. 

// 4732 — account added to a privileged local group (privilege escalation)
SecurityEvent
| where EventID == 4732
| where TargetAccount contains "Administrators"
| project TimeGenerated, Computer, SubjectAccount, MemberName, TargetAccount
| extend MitreTactic = "Privilege Escalation", MitreTechnique = "T1078 - Valid Accounts"

// 4719 — audit policy changed (defense evasion / log tampering)
SecurityEvent
| where EventID == 4719
| project TimeGenerated, Computer, SubjectAccount, AuditPolicyChanges
| extend MitreTactic = "Defense Evasion", MitreTechnique = "T1562.002 - Disable Windows Event Logging"

// 4698 — scheduled task created (persistence)
SecurityEvent
| where EventID == 4698
| project TimeGenerated, Computer, SubjectAccount, TaskName
| extend MitreTactic = "Persistence", MitreTechnique = "T1053.005 - Scheduled Task"

Three post-compromise detections the base tutorial never reaches for.

With the queries written, turning one into an alert is a couple of clicks: New alert rule → Create Microsoft Sentinel rule. Name it, set a severity, and — the genuinely nice part — tag it with its MITRE tactic right in the rule UI. For the brute-force success rule that's Initial Access; for the scheduled-task rule it's Persistence. Set how often it runs (anywhere from five minutes to fourteen days) and save.

Where I tightened it#

It's worth laying out plainly what changed between the tutorial's setup and the one I ended up running, because most of it is the difference between a demo and something that resembles a working detection environment.

AreaBase tutorialWhat I ran instead
Detection logicString match on "success"Event-ID based (4624 / 4625 / 4719 / 4732 / 4698) with thresholds
SeverityOne level for everythingTiered: brute-force Medium, post-brute success High, policy change Critical
MITRE mappingSingle tactic in the UITactic + technique embedded in the KQL so it rides into the incident
Lookback window5-min run, 5-min lookback5-min run, 1-hour lookback — catches slow campaigns under the per-window bar
Entity mappingNot configuredIP and Account mapped as entities — enables the investigation graph and UEBA
Incident groupingDefault (one per alert)Grouped by IP + 1-hour window — no alert storm from a single attacker
AutomationNoneLogic App playbook: geo + AI enrichment on incident creation

The incident-grouping change is the one that saved my sanity. A single determined IP can trip a brute-force rule hundreds of times in an hour. Grouped by source IP and a time window, that becomes one incident with a count, instead of a screen full of identical noise that buries anything real.

The AI layer#

This is where the project stops being a lab build and starts being something I'd actually want at a SOC desk. The most tedious part of Tier-1 work is reading raw event data and reconstructing, in your head, the story of what happened. That reconstruction is exactly the kind of thing a language model is good at — so I put one in the pipeline.

When Sentinel creates an incident, a Logic App fires immediately. It pulls the alert entities — IP, account, timestamps, event IDs — enriches the IP with geolocation, and hands the whole bundle to a model with a tight, structured prompt. The result gets posted back as an incident comment before an analyst opens it. The payload the Logic App sends looks like this: 

{
  "model": "gpt-4o",
  "messages": [
    {
      "role": "system",
      "content": "You are a SOC analyst assistant. Given a Microsoft Sentinel incident, produce a triage summary in three sections: 1) What happened (2-3 sentences), 2) MITRE ATT&CK mapping with technique IDs, 3) Recommended immediate actions. Write for a Tier 2 analyst."
    },
    {
      "role": "user",
      "content": "Incident: @{triggerBody()?['object']?['properties']?['title']}\n\nEntities:\n@{variables('EntitySummary')}\n\nGeo:\n@{variables('GeoEnrichment')}"
    }
  ],
  "max_tokens": 500,
  "temperature": 0.1
}

The Logic App's HTTP action body. Low temperature keeps the output factual and consistent.

For testing the prompt outside Logic Apps — which I'd recommend before you wire it into a playbook — the same call in Python against Azure OpenAI is straightforward: 

import os
from openai import AzureOpenAI

client = AzureOpenAI(
    api_key=os.environ["AZURE_OPENAI_KEY"],
    api_version="2024-02-01",
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
)

def triage(incident_title: str, entities: dict) -> str:
    """Generate an analyst-ready triage summary for a Sentinel incident."""
    summary = "\n".join([
        f"- IP: {entities.get('ip', 'N/A')}",
        f"- Account: {entities.get('account', 'N/A')}",
        f"- Host: {entities.get('host', 'N/A')}",
        f"- Country: {entities.get('country', 'N/A')} ({entities.get('asn', 'N/A')})",
        f"- Failed attempts: {entities.get('failed', 0)}",
    ])

    resp = client.chat.completions.create(
        model="gpt-4o",                       # your Azure deployment name
        messages=[
            {"role": "system", "content": (
                "You are a SOC analyst assistant. Produce a triage summary with "
                "three sections: What happened (2-3 sentences), MITRE ATT&CK mapping "
                "(tactic + technique ID), and three recommended actions. Be technical "
                "and precise; assume a Tier 2 audience."
            )},
            {"role": "user", "content": f"Incident: {incident_title}\n\nEntities:\n{summary}"},
        ],
        max_tokens=500,
        temperature=0.1,
    )
    return resp.choices[0].message.content


if __name__ == "__main__":
    print(triage(
        "Brute Force Success — RDP (T1110.001)",
        {"ip": "185.220.101.47", "account": "Administrator", "host": "honeypot-vm-01",
         "country": "Netherlands", "asn": "Tor exit node", "failed": 347},
    ))

The same triage call as a standalone script, useful for iterating on the prompt.

What comes back is the thing an analyst would otherwise spend four to six minutes assembling by hand: a plain statement of what happened, the technique IDs, and the next moves. For the example above the model produced a summary that read, in essence, that a Tor-exit IP in the Netherlands had run 347 failed logons against the Administrator account before a successful authentication, mapped it to T1110.001, and recommended isolating the host, rotating the credential, and checking for the persistence event IDs I wired up earlier. That last part is the payoff — the AI's advice loops back to the exact detections in this lab.

The second AI touch is enrichment. A raw IP is almost useless to an analyst, so before the triage call I run the IP through a geo/ASN lookup and a Tor/VPN/datacenter classification, then let the model interpret what that combination implies about the actor. A residential IP in one country reads very differently from a datacenter range or a known Tor exit, and surfacing that distinction up front changes how you triage. 

import requests

def enrich_ip(ip: str, client) -> dict:
    """Geo + ASN lookup, then an AI read on what the combination suggests."""
    geo = requests.get(
        f"http://ip-api.com/json/{ip}"
        "?fields=country,regionName,isp,org,as,proxy,hosting",
        timeout=5,
    ).json()

    context = (
        f"IP: {ip}\nCountry: {geo.get('country')}\n"
        f"Org: {geo.get('org', geo.get('isp'))}\nASN: {geo.get('as')}\n"
        f"Proxy/VPN: {geo.get('proxy')}\nHosting/DC: {geo.get('hosting')}"
    )

    read = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": (
                "You are a threat-intel analyst. In 2-3 sentences, assess likely "
                "actor type (opportunistic scanner, targeted, botnet node), "
                "infrastructure type, and OPSEC posture. Be concise."
            )},
            {"role": "user", "content": context},
        ],
        max_tokens=200, temperature=0.2,
    )
    return {"geo": geo, "assessment": read.choices[0].message.content}

IP enrichment feeding an AI actor-context read, which then feeds the triage prompt.

The third piece, which I kept deliberately conservative, is response. The Logic App can ask the model which containment action fits — isolate the host, block the IP at the NSG, watchlist it, or do nothing — and then it sends that recommendation to me for approval before anything executes. The model advises; a human still pulls the trigger. For a lab that's slightly theatrical, but it's the right shape for the real thing, and it means the automation never takes an irreversible action on its own.

What actually showed up#

Then I turned it on and waited, which did not take long. The first brute-force attempt against the box landed inside fifteen minutes of it going public — internet-wide scanners find a fresh public IP almost immediately, and there is something genuinely sobering about watching the first failed-logon events roll in before you've finished your coffee.

Over the next day the failed-logon traffic came from dozens of countries, with the usual suspects dominating the volume: hosting ranges, Tor exits, and a long tail of clearly compromised residential hosts being used as relays. The targeted usernames were exactly the boring, predictable set — Administrator, admin, user, test, guest — because automated tools cycle through the obvious ones first, every time. Volume peaked overnight in my timezone, which lines up with automated campaigns running during business hours elsewhere in the world.

The two things I'll keep from running this: event-ID detections produced zero false positives from the benign logoff events that the tutorial's string-match would have flagged, and the AI triage comment turned each incident from a four-to-six-minute manual read into something I could grasp in well under a minute, because the narrative was already sitting there when I opened it. Neither of those is a dramatic result on its own. Together they're the difference between a lab and a workflow.

The cost footnote I promised. Sentinel bills on data ingestion, and a wide-open VM generating thousands of events an hour with "All Security Events" selected adds up faster than you'd guess. I ran a smaller VM than recommended to throttle event volume, kept it deallocated whenever I wasn't actively collecting, and deleted the whole resource group when I was done. Total cost across about three days of collection: a few dollars. Leave it running and forget it, and that number stops being funny.

Where this goes next#

This is the first piece of a longer cloud-security thread I'm working through. The geo data I'm already collecting wants to become a live attack map in a Sentinel Workbook — watching the failed logons light up a world map in real time is the obvious next move, and the enrichment is half-built for it already. After that I want to wire MISP into Sentinel so attacker IPs get cross-referenced against known threat-actor infrastructure automatically, and layer Defender for Cloud on top to capture what actually happens on the box after a brute-force succeeds, rather than inferring it from event IDs.

The thing this lab drove home, though, is smaller and more durable than any of that. You can read about RDP brute-force for years and it stays abstract. Open one real port to the internet, watch the attempts arrive inside fifteen minutes, and it stops being abstract permanently. Build the detection around what you watched, not around what the tutorial told you to match, and the queries get sharper because you've seen the thing they're meant to catch.
